Upcoming Features

General Features

Note History

It would be useful to be able to refer to a history of recent notes in some cases, one can imagine an action where the previous two notes are checked for their duration before deciding on an action. This history would likely be limited to 3-5 notes, and could possibly also extend into the future so that future notes could be checked.

Additional CSV Notemapper Attributes/Files

More attributes can be added to the CSV Notemappers, additionally, more CSV Notemappers of different instruments can be added as the ones existing now are specifically for the violin.

Tool for CSV Notemapper generation

The way to build your own CSV Notemapper files currently involves writing your own code to do so. This is not a long term solution and in the future a tool which is easier to use should be developed for this purpose. It is currently not clear how this tool might look, but it should be intuitive and usable for people with little to no programming experience.

Improved Output

The three types of output currently are:

  • Basic Output:Partially direct output from the code itself, less useful for regular users.
  • CSV Output:Output in CSV format, useful if the output is to be parsed in another program later on.
  • Readable Output:Output designed to be read by humans with the intent of using the program.

A fourth option which would be useful would be a graphical output which could potentially be overlayed onto the original score. This could either be a direct output of the program or a another application built upon CSV files outputted by this program.

GUI Features

DSL Menu selections

Rather than having to write out most parts of the DSL there could be a menu selection of those parts. This would be for selections with limits, such as which action type is used, or which attribute is used in actions (the attributes themselves would be typed in the attributes part, but then the selection would give you a list of pre-defined attributes.).

DSL Features

DSL Typechecker

Any errors or mistakes in writing the DSL will only be found at runtime, and these errors can be difficult to locate with the exception messages alone. A typechecker for the DSL should be developed to check the code in the DSL file window in the GUI, and possibly with an option in the CLI to check that the DSL is without errors.
